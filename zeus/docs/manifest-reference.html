Manifest Reference

Contract for zeus.plugin.json (schema version 1).

Required Top-Level Fields

  • schema_version: must be 1 (integer)
  • id: slug-safe plugin id
  • name, version, description
  • components: at least one of client or server
  • config_schema: object with client and server field maps

Component Contract

FieldTypeRules
components.client.runtimestringMust be python.
components.client.entrystringRequired when client component exists. Relative .py path.
components.server.runtimestringMust be python.
components.server.entrystringRequired when server component exists. Relative .py path.
components.server.channelsarray<string>Optional channel names.

Config Schema Rules

  • Define fields under config_schema.client and/or config_schema.server.
  • Each field requires type and secret (boolean).
  • Allowed type: string, integer, number, boolean, object, array.
  • Use required: true to enforce user input during the single Install flow.
  • For secret fields, values are collected during Install, persisted in Zeus secret vault, and stored in plugin state only as secret refs.

Secret Storage

V1 supports vault-only storage.

"secret_storage": "vault"

# or
"secret_storage": {
  "default": "vault",
  "fields": {
    "client.api_key": "vault",
    "server.verify_token": "vault"
  }
}

Prerequisites and Zeus Install Handoff

Declare prerequisites in a structured way so Zeus can open a notch install thread, create one todo per missing prerequisite, and guide the user through installation before the next Install step.

"prerequisites": {
  "agent_guidance": "Install each CLI one at a time. Auto-run only declared shell commands.",
  "cli": [
    {
      "id": "google_workspace_cli",
      "name": "Google Workspace CLI",
      "binary": "gws",
      "required": true,
      "why": "Runs Gmail and Workspace operations for this plugin.",
      "check_command": "gws --version",
      "install_options": [
        {
          "type": "shell",
          "label": "Install with npm",
          "description": "Installs the CLI globally with npm.",
          "command": "npm install -g @googleworkspace/cli",
          "auto_run": true,
          "requires_admin": false,
          "platforms": ["darwin", "linux"]
        },
        {
          "type": "open_url",
          "label": "Open install docs",
          "url": "https://github.com/googleworkspace/cli"
        }
      ]
    }
  ],
  "environment_check": {
    "client_function": "check_environment"
  }
}
  • Every prerequisite must define a detection path with binary and/or check_command.
  • install_options[].type supports shell and open_url in v1.
  • shell options can be auto-run only when auto_run: true and requires_admin: false.
  • Older manifests that only declare auto_install_commands, install_hint, and install_url are still normalized into this model for compatibility.

OAuth Modes

  • mode: providers: OAuth provider metadata model.
  • mode: cli_sequence: Zeus runs explicit CLI commands for plugin auth. Use interactive: true when auth must stream CLI output, pause for browser/manual work, and resume in the notch.
  • env: optional map of ENV_VAR_NAME to client.field or server.field for non-secret config values. Zeus injects optional bindings only when those fields are set.
  • secret_env: optional map of ENV_VAR_NAME to a secret config field path. Zeus resolves the Secret Vault key at runtime, never prints the secret value in the handoff prompt, and injects optional bindings only when those fields are set.
  • mode: none: no OAuth actions.
"oauth": {
  "mode": "cli_sequence",
  "login_command": "gws auth login",
  "interactive": true,
  "success_check_command": "gws auth status",
  "secret_env": {
    "GOOGLE_WORKSPACE_CLI_CLIENT_ID": "client.google_client_id",
    "GOOGLE_WORKSPACE_CLI_CLIENT_SECRET": "client.google_client_secret"
  },
  "description": "Run gws auth through the Zeus install handoff",
  "agent_prompt": "Optional instructions shown to Zeus during install"
}
  • setup_command and login_command are both optional individually, but at least one must be present.
  • Use interactive: true when the command may open a browser, print URLs, or require user work outside Zeus.
  • When interactive: true, declare success_check_command so Zeus can deterministically verify completion after the agent handoff finishes.
  • If an env or secret_env binding points at a required config field, Install still blocks until that required field is provided.

Shipped Skills

Declare bundled skills for automatic install/remove with plugin lifecycle.

"skills": {
  "bundled": true,
  "path": "skills",
  "auto_install": true,
  "auto_enable": true,
  "uninstall_policy": "remove"
}

Lifecycle Notes

  • New installs are inactive until prerequisites, configuration, and auth are all ready.
  • prerequisite_state covers environment readiness: not_required, pending, running, succeeded, failed.
  • setup_state remains auth-specific (for example OAuth CLI sequences).
  • The app exposes one Install action. Zeus runs prerequisite install first, opens the settings/secret form when configuration is still needed, and then continues into auth.
  • Interactive auth uses the same notch-thread handoff: Zeus streams CLI output, lets the user complete browser/manual steps, then rechecks success_check_command before marking Install complete.
  • Plugin can be enabled only when enabled=true, configured=true, prerequisite_state is ready, and setup_state is ready.

Migration Note (Pre-Configure-Modal Plugins)

  • Legacy JSON workflows (/api/plugins/config + direct secret_refs) still exist for compatibility.
  • New plugins should target /api/plugins/configure and provide secrets via secrets payload.
  • Legacy prerequisite fields (auto_install_commands, install_hint, install_url) still load, but new plugins should author install_options[].
  • Keep index.json as the single allowlisted catalog source.
