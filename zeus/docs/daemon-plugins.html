Daemon Plugin Runtime

How Zeus loads and executes plugin-defined tools, operations, and hooks on the Mac daemon.

Load Conditions

  • Plugin exists in local plugins_state.json.
  • Plugin is enabled, configured, and setup state is not_required or succeeded.
  • components.client.runtime is python.

Plugin Object Contract

Client entry module exports register(context) and returns an object that may implement:

  • Operation declarations: tools(), commands(), services(), cli(), providers(), gateway(), http().
  • Operation handlers: call_tool, call_command, call_service, call_cli, call_provider, call_gateway, call_http (or execute_* variants).
  • Hooks: before_model_resolve, before_prompt_build, before_tool_call, after_tool_call.

Hook Phase Behavior

PhaseTypical UseBlockable
before_model_resolveAdjust model/provider selection inputs.No (block ignored).
before_prompt_buildMutate system prompt payload.No (block ignored).
before_tool_callInspect/mutate tool input or prevent execution.Yes.
after_tool_callNormalize/augment tool results.No (block ignored).

Runtime Policy Enforcement

  • runtime_policy.network.socat_bypass applies default network behavior.
  • runtime_policy.approval.mode applies default approval behavior.
  • runtime_policy.tools.{tool}.enabled controls per-tool enablement.
  • runtime_policy.tools.{tool}.allow_network and allow_without_approval override defaults per tool.

Default network posture is stricter for tool/command/service/cli, and more permissive for provider/gateway/http, unless overridden in policy.

Prerequisites and Environment Checks

  • prerequisites.cli[] declares external CLI dependencies (example: gh).
  • Each CLI prerequisite can include install_hint, install_url, and auto_install_commands (darwin/linux).
  • prerequisites.environment_check.client_function points to a Python module function (for example check_environment).
  • The function should return readiness as bool, (bool, message), or { "ok": bool, "message": "..." }.

If prerequisites/environment checks fail, Zeus keeps the plugin installed but does not load runtime operations until checks pass.

Configure + Setup Lifecycle

  • User configures plugin via app modal (Configure Plugin button).
  • Secrets from the form are persisted to secret vault and converted to secret refs automatically.
  • Optional setup runner executes manifest-defined actions (including OAuth CLI sequences).
  • Setup status is persisted in plugin state with user-facing status reason.

Context Passed to Plugin Runtime

{
  "plugin_id": "sample_plugin",
  "plugin_version": "1.0.0",
  "config": {...},
  "secret_refs": {...},
  "secrets": {...},
  "operations": {
    "tools": [...],
    "commands": [...],
    "services": [...],
    "cli": [...],
    "providers": [...],
    "gateway": [...],
    "http": [...]
  }
}

Secrets are resolved from Zeus secret vault using references; plugins receive resolved values only at runtime context.
