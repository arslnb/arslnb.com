OpenClaw Differences

Zeus plugins are loosely inspired by OpenClaw concepts, but are Zeus-native and Python-first.

Shared Ideas

  • Manifest-driven plugin capabilities.
  • Pluggable tools/hooks/channels abstraction.
  • Registry-oriented install and update model.

Key Differences

AreaOpenClaw styleZeus Native
Runtime languageTypeScript/Node-centricPython runtime for both daemon and server components
Artifact contractOpenClaw-specific manifest/host assumptionszeus.plugin.json with Zeus component/policy/OAuth/automation fields
Execution surfacesTool/plugin host-centricDual runtime: local daemon + server channel runtime
Channel modelDepends on OpenClaw host APIsGeneric /v1/channels/{channel}/webhook, /v1/client/channels/{channel}/send, and /webhook/{route}
Install activationHost dependentControlled in-process reload with rollback on failure
Security/authHost dependentSigned device-authenticated server control endpoints + curated registry verification
Policy controlsVaries by hostManifest-level runtime policy including tool enablement/network/approval defaults
OAuth flowsPlugin-host specificGeneric OAuth provider declarations linked to config schema and secret refs
Proactive + scheduled promptsNot standardized across hostsManifest-native automations and proactive sections

Porting Guidance

  1. Map OpenClaw manifest metadata into zeus.plugin.json.
  2. Re-implement runtime code in Python entry modules (client/plugin.py and/or server/plugin.py).
  3. Translate tool-level permissions into Zeus runtime_policy.
  4. Replace host-specific webhook routing with Zeus channel/custom webhook contracts.
  5. Declare OAuth providers in manifest and move secrets to secret_refs.

Compatibility Expectation

Zeus does not treat OpenClaw packages as drop-in binaries. Migration is source-level porting to Zeus runtime contracts. Once ported, plugin behavior is managed entirely through Zeus APIs and policy controls.
