Channel Plugin Runtime

How server plugins expose channels, custom webhooks, and outbound send operations.

Server Plugin Contract

  • Manifest includes components.server.entry with runtime python.
  • Server module exports register(ctx).
  • Returned plugin object can implement channels() and/or webhooks().
  • ctx now includes server config, secret refs, resolved secret values (ctx.secrets), and connected server OAuth sessions for that plugin.

Channel Adapter Interface

MethodRequiredPurpose
verify_webhook(request)NoHandle provider verification requests (typically GET).
ingest_webhook(...)Yes (for inbound)Parse provider payload and return list of envelopes.
send(payload)Yes (for outbound)Send outbound message and return provider result.

channels() can return either {channel_name: adapter} or an ordered list aligned with components.server.channels.

Custom Webhook Interface

Plugins can expose arbitrary provider routes through webhooks():

def webhooks(self):
    return {
      "stripe-events": {
        "methods": ["POST"],
        "handler": self.handle_stripe
      },
      "calendar-push": self.handle_calendar_push
    }
  • "stripe-events" maps to /webhook/stripe-events.
  • Allowed methods can be explicit or omitted (defaults to all methods).

Runtime Routes

  • GET /v1/channels/{channel}/webhook -> verify_webhook
  • POST /v1/channels/{channel}/webhook -> ingest_webhook
  • POST /v1/client/channels/{channel}/send -> send (signed device auth)
  • ANY /webhook/{route} -> custom plugin webhook handler

Event Stream Behavior

Plugin channel/webhook handlers can return message envelopes (or return values convertible to envelopes). Zeus normalizes output and emits it onto the same secure client event stream consumed by standard message APIs.

If you want agent-driven replies to route back through the same channel automatically, include a provider-specific metadata.reply_context object on inbound envelopes. Zeus uses that reply context when the Mac app sends the final assistant response back through POST /v1/client/channels/{channel}/send.

Channel plugins should generally avoid provider-specific slash commands and instead let inbound messages flow through the normal agent loop. The live Telegram reference plugin follows that pattern.

Live Reference Plugin

The registry sample kitchen_sink_plugin now exercises this contract against a real Telegram Bot API channel and a real GitHub OAuth provider. Use it as the reference when you want a working integration example, not just a shape-only fixture.

Install and Activation Semantics

  • Install/update/enable/config goes through /v1/client/plugins/*.
  • Server performs controlled in-process reload.
  • On reload failure, previous plugin/channel map is restored.
  • On success, routes become available immediately.
  • If ZEUS_PUBLIC_BASE_URL is set, plugins can auto-register provider webhooks during on_enable().
