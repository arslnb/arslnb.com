Channel Plugin Runtime

How server plugins expose channels, custom webhooks, and outbound send operations.

Server Plugin Contract

  • Manifest includes components.server.entry with runtime python.
  • Server module exports register(ctx).
  • Returned plugin object can implement channels(), webhooks(), and/or signed actions().
  • ctx now includes server config, secret refs, resolved secret values (ctx.secrets), and connected server OAuth sessions for that plugin.

Channel Adapter Interface

MethodRequiredPurpose
verify_webhook(request)NoHandle provider verification requests (typically GET).
ingest_webhook(...)Yes (for inbound)Parse provider payload and return list of envelopes.
send(payload)Yes (for outbound)Send outbound message and return provider result.

channels() can return either {channel_name: adapter} or an ordered list aligned with components.server.channels.

Custom Webhook Interface

Plugins can expose arbitrary provider routes through webhooks():

def webhooks(self):
    return {
      "stripe-events": {
        "methods": ["POST"],
        "handler": self.handle_stripe
      },
      "calendar-push": self.handle_calendar_push
    }
  • "stripe-events" maps to /webhook/stripe-events.
  • Allowed methods can be explicit or omitted (defaults to all methods).

Server Action Interface

Plugins can also expose signed server-side actions for daemon tools or UI flows that need server-only state such as OAuth tokens:

def actions(self):
    return {
      "github_me": self.github_me,
      "github_notifications": self.github_notifications,
    }

Zeus invokes these through POST /v1/client/plugins/{plugin_id}/actions/{action} using the same signed device-auth headers as the rest of the plugin control plane.

Runtime Routes

  • GET /v1/channels/{channel}/webhook -> verify_webhook
  • POST /v1/channels/{channel}/webhook -> ingest_webhook
  • POST /v1/client/channels/{channel}/send -> send (signed device auth)
  • POST /v1/client/plugins/{plugin_id}/actions/{action} -> server plugin actions() (signed device auth)
  • ANY /webhook/{route} -> custom plugin webhook handler

Event Stream Behavior

Plugin channel/webhook handlers can return message envelopes (or return values convertible to envelopes). Zeus normalizes output and emits it onto the same secure client event stream consumed by standard message APIs.

If you want agent-driven replies to route back through the same channel automatically, include a provider-specific metadata.reply_context object on inbound envelopes. Zeus uses that reply context when the Mac app sends the final assistant response back through POST /v1/client/channels/{channel}/send.

If your provider supports lightweight native affordances while the full reply is still being generated, also include metadata.processing_context. Zeus forwards that payload immediately through the same signed send route so channels can show typing indicators, reactions, drafts, or similar provider-native acknowledgements.

Canonical threading is keyed by channel + conversation key. A stable channel implementation must emit enough normalized identity data for Zeus to keep one remote conversation mapped to one canonical app thread across follow-up messages.

If a channel plugin needs user-specific identity details to decide whether a message came from the primary owner or from somebody else, declare those config fields in config_schema and expose them to the Mac UI with ui_hints.messaging_channels.<channel>.primary_user_identity. Zeus renders those fields in Settings > Messaging Channels > plugin, but the plugin still owns the matching logic.

When a message is classified as coming from the primary owner, Zeus handles it with the normal prompt/tool surface used in the app. When a message is classified as coming from somebody else, Zeus switches to a restricted executive-assistant mode with no tools and replies as a relay for the owner.

If the user enables Offline replies while Mac is offline in Zeus, the message server can also send a lightweight no-tools fallback reply when the Mac does not claim the turn in time. That fallback uses the same canonical channel + conversation key, must explicitly say the Mac is offline, and should only coordinate or gather detail until the Mac reconnects.

Channel plugins should generally avoid provider-specific slash commands and instead let inbound messages flow through the normal agent loop. The live Telegram reference plugin follows that pattern.

Live Reference Plugin

The registry sample kitchen_sink_plugin now exercises this contract against a real Telegram Bot API channel, a real GitHub OAuth provider, and signed server actions that power the daemon-side github_me and github_notifications tools. Use it as the reference when you want a working integration example, not just a shape-only fixture.

Install and Activation Semantics

  • Install/update/enable/config goes through /v1/client/plugins/*.
  • Server performs controlled in-process reload.
  • On reload failure, previous plugin/channel map is restored.
  • On success, routes become available immediately.
  • If ZEUS_PUBLIC_BASE_URL is set, plugins can auto-register provider webhooks during on_enable().
