Publish to the Zeus Plugin Registry

Once your plugin works locally, open a PR to a shared registry so the community can use it.

Should This Be a Separate Repo?

Yes. Keep plugin registry submissions in a dedicated repository: arslnb/zeus-plugins.

  • Cleaner trust boundary: plugin submissions are reviewed separately from Zeus core.
  • Faster iteration: registry metadata and plugin updates can ship independently.
  • Simpler governance: contributors only need registry access patterns, not core repo internals.
  • Better discoverability: one canonical place for all public plugin listings.

Submission Mechanism

Use a standard pull-request flow against arslnb/zeus-plugins:

  1. Contributor forks the registry repo and creates a branch.
  2. They add one folder under plugins/<plugin_id>/ with manifest, metadata, and docs.
  3. They open a PR using a required PR template.
  4. CI validates JSON schema, required fields, and file presence.
  5. On merge, Zeus reads index.json and exposes new plugins to all users.

Use Zeus CLI for Scaffold + Validation

git clone https://github.com/arslnb/zeus-plugins.git
cd zeus-plugins
python3 -m venv .venv
source .venv/bin/activate
python -m pip install -e .

# Generate a plugin starter
zeus plugin init acme_demo --runtime daemon --owner acme

# Check a plugin (machine-readable)
zeus plugin check plugins/acme_demo --json

# Check full registry before PR
zeus plugin check . --registry --json

Tasks:
1) Validate zeus.plugin.json and ensure runtime entries are real.
2) Add metadata.json and index.json entry.
3) Add install/config examples and a short README.
4) Add a PR summary with:
   - capability overview
   - required secrets/config
   - security notes
   - test commands and output
5) Generate the exact git/gh commands needed to open the PR.

Return:
- full file tree for the registry submission
- complete file contents
- copy-paste commands to open a PR.

Metadata Format (Suggested)

{
  "id": "acme_demo",
  "name": "Acme Demo",
  "version": "1.0.0",
  "summary": "Looks up customer records and syncs notes",
  "owner": "acme",
  "runtime": ["client", "server"],
  "categories": ["support", "crm"],
  "min_zeus_version": "1.0.0",
  "source_repo": "https://github.com/acme/zeus-acme-plugin",
  "license": "MIT"
}

CI Validation Gate (Suggested)

name: validate-registry
on:
  pull_request:
    paths:
      - "plugins/**"
      - "index.json"
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: python scripts/validate_registry.py

How Zeus Should Read the Registry

Keep a root index.json containing all published plugins with stable IDs and source refs.

[
  {
    "id": "acme_demo",
    "version": "1.0.0",
    "metadata_path": "plugins/acme_demo/metadata.json",
    "manifest_path": "plugins/acme_demo/zeus.plugin.json"
  }
]

Zeus can sync this index on boot and periodically (or on webhook) to make new plugins globally discoverable.

PR Workflow

1) Fork + branch

git clone https://github.com/arslnb/zeus-plugins.git
cd zeus-plugins
git checkout -b add-acme-demo-1-0-0

2) Add plugin files

mkdir -p plugins/acme_demo/examples
# copy zeus.plugin.json, metadata.json, README, examples/*

3) Validate and commit

git add plugins/acme_demo
git commit -m "Add acme_demo plugin v1.0.0"

4) Push and open PR

git push origin add-acme-demo-1-0-0
gh pr create --repo arslnb/zeus-plugins \
  --title "Add acme_demo plugin v1.0.0" \
  --body "Adds acme_demo plugin with install/config examples and metadata."

Review Checklist

  • Manifest parses and declared runtime entry files exist.
  • Required config and secret refs are documented.
  • Example install/config payloads are valid JSON.
  • No unsafe defaults (network, approvals, unrestricted tool surface).
  • Plugin version and changelog are consistent.
