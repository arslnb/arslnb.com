Publish a Plugin

Submit your plugin to the approved Zeus catalog so users can install it from Settings → Plugins.

Catalog Policy (Authoritative Allowlist)

  • index.json is the definitive approval list.
  • The app installs/searches only plugins listed in index.json.
  • Plugin folders not listed in index.json are ignored by catalog validation and app discovery.
  • Catalog installs track the repository main branch.

Quick Links

Static Registry Publish (For Server Installs)

The GitHub repo is the source of truth, but the Zeus message server does not install plugins by cloning GitHub. It installs from a signed static registry.

cd zeus-plugins
python -m pip install -e .

# generate once
zeus plugin keygen --out-dir .registry-keys

# publish approved plugins to a static directory
zeus plugin publish . \
  --output-dir dist/registry \
  --base-url https://plugins.example.com \
  --private-key .registry-keys/ed25519-private.pem \
  --clean

Host dist/registry/ on any static host. Zeus expects:

  • index.json at the registry root for the app catalog
  • plugins/<plugin_id>/<version>.json for server installs
  • artifacts/<plugin_id>/<version>/<plugin_id>-<version>.tgz for the packaged plugin

Then set on the Zeus message server:

  • ZEUS_PLUGIN_REGISTRY_BASE_URL=https://plugins.example.com
  • ZEUS_PLUGIN_REGISTRY_PUBLIC_KEY=<contents of dist/registry/registry-public-key.b64>

For github.com/arslnb/zeus-plugins, the simplest path is GitHub Pages. This repo now includes a workflow that publishes dist/registry/ to Pages on every push to main.

  • Required secret: ZEUS_REGISTRY_SIGNING_PRIVATE_KEY
  • Optional repository variable: ZEUS_PLUGIN_REGISTRY_BASE_URL for a custom domain
  • Default registry base URL: https://arslnb.github.io/zeus-plugins

1) Scaffold and Validate

git clone https://github.com/arslnb/zeus-plugins.git
cd zeus-plugins
python3 -m venv .venv
source .venv/bin/activate
python -m pip install -e .

# create scaffold
zeus plugin init acme_demo --runtime daemon --owner acme

# validate plugin folder
zeus plugin check plugins/acme_demo --json

# validate allowlisted catalog
zeus plugin check . --registry --json

2) Required Submission Files

plugins/
  acme_demo/
    zeus.plugin.json
    metadata.json
    README.md
    examples/
      install.json
      config.json
    client/plugin.py   # and/or server/plugin.py
    skills/            # optional shipped skills
  • Manifest id, folder name, and metadata id must match.
  • Declare prerequisite install in manifest (prerequisites.agent_guidance, prerequisites.cli[].install_options[]) where applicable.
  • Declare auth separately in oauth; prerequisite install and auth are different phases inside the same Install flow.
  • Use oauth.secret_env when auth commands need vault-backed credentials during Install. Optional bindings are injected only when configured.
  • For CLI auth flows that may open a browser or require manual console work, set oauth.interactive=true and provide oauth.success_check_command.
  • Declare shipped skills in manifest skills if you bundle them.

3) Add to Allowlist Index

Add (or update) a single entry in index.json:

{
  "id": "acme_demo",
  "name": "Acme Demo",
  "version": "1.0.0",
  "summary": "One-line summary.",
  "owner": "acme",
  "runtime": ["client"],
  "categories": ["productivity"],
  "min_zeus_version": "1.0.0",
  "source_repo": "https://github.com/arslnb/zeus-plugins",
  "license": "MIT",
  "metadata_path": "plugins/acme_demo/metadata.json",
  "manifest_path": "plugins/acme_demo/zeus.plugin.json"
}

4) Open Pull Request

git checkout -b add-acme-demo-1-0-0
git add plugins/acme_demo index.json
git commit -m "Add acme_demo plugin v1.0.0"
git push origin add-acme-demo-1-0-0

gh pr create --repo arslnb/zeus-plugins \
  --title "Add acme_demo plugin v1.0.0" \
  --body "Adds acme_demo plugin with single-install-flow examples and docs."

Submission Checklist

  • Manifest and metadata pass validation.
  • Index entry is present and paths are correct.
  • README explains the single Install flow, Zeus handoff behavior, and configuration form fields.
  • Examples include /api/plugins/install and /api/plugins/configure.
  • If secrets are needed, docs describe vault-backed configuration.
  • If prerequisites are required, provide at least one install option or docs URL for each required prerequisite.
After merge, users can install from the in-app approved catalog. If prerequisites are missing, Zeus opens a notch install thread, completes the manifest-declared prerequisite steps, collects any required settings and secrets into vault, and then continues into auth. When a plugin also has a server component, Zeus resolves the local vault-backed secret fields and mirrors them into the server's vault during install and Resync Server, so channel sends do not depend on local-only secret references. See the GWS plugin for the reference implementation.
